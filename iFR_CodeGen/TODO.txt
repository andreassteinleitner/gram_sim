List of all open tasks for the IFR CodeGeneration V3 for PX4 Firmware v1.13.2

Known Problems:

	1) SIMULINK cannot create C-Code from model files (.slx) directly working in WSL directory (even when working on a network drive)
		- Currently this is circumvented by building in folder on the local hard drive and moving the necessary files afterwards.
	2) Code Generation crashes when uORB-Messages have sloppy code style. Specifically when there are no spaces around "=".
		- Example:
			Correct: 	uint8 TASK_ONE = 1
			Will crash: uint8 TASK_ONE= 1
	3) Code generation did not work properly when used during the development of the INDI controller with Philipp Julier
		- a problem occurred while trying to subscribe to a certain topic (actuator_controls_0) which has multiples
		- as this happened some time ago the exact details are not known anymore
	4) Code Generation experienced some problems in 2022 with similar named simulink models.
		- The problem occurred with formation_controller_2022 being present in the main simulink_controller folder and a formation_controller_2021 being present in Archives
		- as this happened some time ago the exact details are not known anymore
		- the problem was likely related to selecting the correct parameter busses, if remembered correctly
	5) Code Generation experienced some problems in 2022 when multiple modules had been added to the PX4 Firmware
		- building of PX4 Firmware failed because of similar function names
		- this usually happened when different modules had identical constants (same name and same value) present
		- source of error is Simulink Coder, which does not use really unique names for variables and functions
		- used workaround: slightly change name and value of constants
	6) After opening a model with openModel.m saving after changes have been made may take a very long time
		- caused by working within WSL with certain systems
		- can also be experienced when renaming files on wsl
